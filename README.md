# 洛基之影（Shadow of Loki）

[`my-neuro`](https://github.com/morettt/my-neuro) 的 `live-2d` 社区插件，面向“游戏陪玩 / 游戏攻略辅助”场景设计。

它会结合截图和对话识别游戏上下文，记录当前任务、区域、Boss、章节与角色状态，并在需要时优先检索本地攻略库，再从多平台来源补充信息，输出更贴近当前进度的辅助回答。

> 当前仓库为公开发布版：已移除本地私有路径、API Key、个人隐私信息，以及不适合公开分发的敏感提示内容。

## 功能概览

- **游戏状态追踪**：通过 [`loki_shadow_track`](./index.js) 记录任务、地图、Boss、章节、角色等上下文信息。
- **攻略/剧情查询**：通过 [`loki_shadow_query`](./index.js) 查询当前游戏相关攻略、剧情、配装、Boss 打法等内容。
- **五源并行检索**：支持游民星空、B站、TapTap、NGA、米游社等来源并行获取信息。
- **本地攻略库优先**：优先搜索仓库内 `game-guides/` 社区攻略库，降低重复抓取和重复整理成本。
- **双模型容灾**：支持主 Agent 与后备 Agent 切换。
- **会话增强搜索**：结合近期游戏状态，自动优化低质量查询词。
- **防剧透式辅助**：适合做“陪玩型”辅助，而不是简单硬贴原文攻略。

## 游戏攻略库分享计划

为了让插件越用越强，这个仓库同时维护一个社区共建的游戏攻略库：[`game-guides/`](./game-guides)

### 共享原则

- 攻略请使用 `.txt` 格式上传。
- 一份攻略对应一个文件，文件名尽量写清游戏名、任务名、章节名、Boss 名或主题关键词。
- 欢迎补充主线流程、支线任务、Boss 打法、角色养成、配队思路、版本活动、剧情解析等内容。
- 内容越清晰，插件按文件名筛选时命中率越高。
- 上传之后，你贡献的内容会变成公共攻略库的一部分，帮助后来的使用者。
- 人人为我，我为人人。

### 推荐命名方式

```text
游戏名_任务或章节_主题.txt
游戏名_Boss名_打法攻略.txt
游戏名_角色名_养成配队.txt
```

例如：

```text
鸣潮_曙光停摆于荒地之上_主线攻略.txt
绝区零_第2季第6章_剧情解析.txt
原神_无相之雷_打法攻略.txt
```

### 腾讯文档收集说明文案

下面这段可以直接放进腾讯文档里：

```text
游戏攻略库分享计划

欢迎把你整理好的游戏攻略上传到我们的共享攻略库，一起把游戏陪玩插件喂得更聪明。

上传要求：
1. 攻略文件统一使用 txt 格式。
2. 一份攻略一个文件，文件名尽量写清游戏名、章节、任务、Boss 或主题。
3. 内容可以是主线流程、支线任务、Boss 打法、角色养成、配队思路、活动攻略、剧情整理等。
4. 尽量保持内容清晰、分段明确，方便后续整理与检索。

你上传的每一份攻略，都会进入公共攻略库，帮助更多玩家在卡关、补剧情、查配队时更快找到信息。

这是一个长期共建计划。
你分享一点，我补充一点，大家一起把攻略库做大。

人人为我，我为人人。
```

## 适用场景

- 用户正在游玩单机 / 二游 / 剧情游戏时，结合截图或聊天内容进行辅助。
- 用户卡关，希望获得不过度剧透的分步引导。
- 用户想讨论当前章节、任务、Boss 或角色，但不希望被未来剧情直接剧透。
- 用户需要将零碎截图信息转成结构化搜索词进行攻略查询。

## 工作流程

插件核心流程由 [`orchestrator.js`](./orchestrator.js) 编排，大致如下：

1. 检测当前游戏（或使用显式传入的游戏名）
2. 扫描本地攻略库
3. 分析已有内容是否足够回答
4. 必要时从多来源并行抓取补充资料
5. 整合结果并生成可复用攻略内容
6. 结合当前会话状态生成最终回答

## 安装方式

将本仓库放入：

```text
live-2d/plugins/community/loki-shadow/
```

然后在 `live-2d` 运行依赖安装：

```bash
npm install axios cheerio
```

如果你的运行环境已经包含这些依赖，可跳过。

## 配置说明

公开版已将敏感配置清空。你需要在插件配置界面自行填写。

### 主要配置项

- **游戏攻略库路径**：默认使用插件目录下的 [`./game-guides`](./game-guides)
- **主智能体配置**：API 地址、模型、温度、最大 Token
- **后备智能体配置**：主模型失败时自动切换
- **各来源搜索数量限制**：用于控制抓取规模
- **内容最大长度**：避免发送给模型的文本过长

### 配置建议

- `guide_library_path`：默认可直接使用仓库内 `game-guides/`，也可以改成你自己的可写目录
- `sub_agent.api_key`：填写主模型 API Key
- `fallback_agent.api_key`：可选，填写后备模型 API Key
- `max_content_length`：根据你的模型上下文能力酌情调整

### 攻略库分享计划与配置的关系

- 默认路径已经指向仓库内的公共攻略库，克隆后可直接使用。
- 你也可以把 `guide_library_path` 改到自己的目录，单独维护私有攻略库。
- 如果你愿意把内容共享出来，推荐把整理好的 `.txt` 攻略再提交回本仓库的 `game-guides/`。
- 公共库越大，首次命中本地攻略的概率越高，重复抓取的次数越少。

## 工具说明

### `loki_shadow_track`

用于记录当前观察到的游戏状态，例如：

- 游戏名
- 当前任务
- 当前 Boss
- 当前区域
- 当前章节
- 相关角色列表

这个工具不会直接搜索攻略，但会增强后续查询质量。

### `loki_shadow_query`

用于发起正式查询，例如：

- `原神 无相之雷 打法攻略`
- `鸣潮 第三章 主线任务流程`
- `绝区零 某角色 配队思路`

建议使用结构化关键词，效果会明显好于零碎截图原文。

## 检索性能说明

当前本地攻略库检索逻辑是：

1. 递归扫描 `txt` 文件
2. 按文件名包含游戏名做初筛
3. 把候选文件名交给模型选择最相关文件
4. 读取命中文件内容

这不是全文索引检索，更接近“文件名筛选 + 内容读取”。

以当前仓库内这批攻略为例：

- 攻略文件数：27
- 总体积：约 486 KB
- 全库扫描平均耗时：约 0.9 ms
- 去重索引首次构建耗时：约 9.1 ms
- 最大单文件读取平均耗时：约 0.33 ms

所以按现在这批规模，几乎感觉不到本地 I/O 压力，实际延迟更多来自模型调用和联网抓取。

但当文件量明显增大时，影响会线性上升：

- **约 1,000 个文件**：本地扫描通常仍在几十毫秒级，影响轻微。
- **约 10,000 个文件**：扫描和候选列表会明显变慢，可能进入百毫秒到秒级区间。
- **真正更大的瓶颈**：候选文件名过多时，需要把更多文件名交给模型选择，模型输入会膨胀，响应时间和成本会上升得比本地读盘更明显。

结论很直接：

- 现在这批攻略量不会明显影响检索效率。
- 后续真要做大规模共建，优先保证文件名规范，其次再考虑轻量索引或按游戏分目录。
